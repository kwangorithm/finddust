🌿 핵심 개념
Branch란 무엇인가?
Branch = 메인 코드를 건드리지 않고 독립적으로 작업할 수 있는 평행 세계
실험이 실패해도 main 브랜치는 안전합니다.
main
C1
C2
C5
C6
← Merge!
C3
C4
feature/pm25-analysis 브랜치
연구에서 Branch 활용 예
main — 최종 완성 코드
feature/pm25 — 새 분석 시도
experiment/ml-model — ML 실험
fix/data-bug — 버그 수정
왜 Branch를 쓰나?
✅ main을 항상 안전하게 유지
✅ 여러 기능을 병렬로 개발
✅ 실험 실패 시 깔끔하게 삭제
04 / 15
⌨️ 실습
Branch 생성 · 전환 · 병합
bash — Branch 기본 워크플로우 
# 현재 브랜치 목록 확인
git branch

# 새 브랜치 생성 + 즉시 전환 (가장 자주 씀)
git switch -c feature/pm25-analysis

# 브랜치에서 작업 후 커밋
git add .
git commit -m "PM2.5 월별 평균 분석 코드 추가"

# main으로 돌아오기
git switch main

# 브랜치 병합 (main에 feature 내용 합치기)
git merge feature/pm25-analysis

# 병합 완료 후 브랜치 삭제 (선택)
git branch -d feature/pm25-analysis
💡 구 명령어 vs 신 명령어
git checkout -b 대신 최신 Git은 git switch -c 를 권장합니다.

⚠️ 충돌 해결
Merge Conflict — 충돌 해결하기
⚠️ Merge Conflict란?
두 브랜치에서 같은 파일의 같은 줄을 다르게 수정했을 때 Git이 어느 쪽을 선택할지 모름 → 사람이 직접 해결해야 함
충돌 파일 내용 예시 (analysis.py) 
<<<<<<< HEAD  (현재 브랜치 - main)
threshold = 35   # WHO 기준
=======
threshold = 15   # 한국 환경부 기준
>>>>>>> feature/pm25-analysis
1
VS Code에서 충돌 파일 열기 → 파란/초록 버튼으로 원하는 버전 선택
2
<<<<<<< ======= >>>>>>> 마커 삭제 후 최종 코드만 남기기
3
git add . → git commit 으로 충돌 해결 완료

🔀 핵심 기능
Pull Request — 코드 리뷰 요청
PR = 브랜치의 변경사항을 main에 합치기 전에 팀원(지도교수)에게 검토를 요청하는 과정
1
브랜치에서 작업 완료 후 GitHub에 push
2
GitHub 저장소 페이지 → Compare & pull request 버튼 클릭
3
제목 + 설명 작성: 어떤 분석을 추가했는지, 어떤 데이터를 사용했는지
4
Reviewers에 지도교수 또는 공동연구자 지정 → Create pull request
5
리뷰 통과 후 Merge pull request → main에 반영 완료 🎉
💡 연구실 활용 팁
지도교수를 Reviewer로 지정하면 코드 변경사항에 줄 단위 코멘트를 받을 수 있어요.

⌨️ 실습
Pull Request 실습
bash — 브랜치 생성 → 작업 → PR 제출 전체 흐름 
# 1. 새 기능 브랜치 생성
git switch -c feature/seoul-pm25-trend

# 2. 파일 수정 후 커밋
git add src/visualize.py
git commit -m "서울 PM2.5 연도별 트렌드 시각화 추가"

# 3. GitHub에 브랜치 push (처음엔 -u 옵션 필요)
git push -u origin feature/seoul-pm25-trend

# → 이제 GitHub 페이지에서 PR 생성!
✅ 좋은 PR 설명 예시
제목: 서울 PM2.5 2015–2023 연도별 트렌드 시각화 추가
설명: AirKorea 데이터 기반 / matplotlib 사용 / Figure 2 초안 / 리뷰 포인트: 색상 scheme 적절한지?

🤝 협업 시나리오
Fork — 공동 연구 협업 흐름
Fork = 다른 사람의 저장소를 내 계정으로 복사해서 자유롭게 수정하는 것
수정 후 원본 저장소에 PR을 보낼 수 있음
지도교수
원본 저장소
→ Fork →
내 계정
복사 저장소
→ 수정 →
PR 제출
원본에 기여
bash — Fork 후 로컬 작업 + 원본 동기화 
# Fork한 내 저장소 클론
git clone https://github.com/내계정/finedust-research.git

# 원본(upstream) 저장소 등록
git remote add upstream https://github.com/교수님계정/finedust-research.git

# 원본의 최신 내용 가져오기
git fetch upstream
git merge upstream/main

오늘 배운 명령어 요약
명령어설명사용 시점
git branch 브랜치 목록 확인 현재 어떤 브랜치인지 확인할 때
git switch -c 이름 브랜치 생성 + 전환 새 기능/분석 시작할 때

git push -u origin 브랜치명 브랜치 GitHub에 올리기 PR 생성 전
git remote -v 원격 저장소 확인 upstream 등록 여부 확인
git fetch upstream 원본 저장소 변경 가져오기 Fork 후 동기화
